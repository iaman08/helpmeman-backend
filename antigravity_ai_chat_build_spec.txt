================================================================
ANTIGRAVITY — AI ASSISTANT FEATURE BUILD SPECIFICATION
================================================================
Version: 1.0
Prepared for: Antigravity Development Team
Feature scope: AI Chat (Menu Drawer Integration + Memory + History + Meeting Summaries)

================================================================
OVERVIEW
================================================================

Replace the existing AI pop-up chat window with a full-featured
AI Assistant accessible from the main menu drawer. The new system
includes persistent memory, resumable chat history, automatic
meeting summaries, date-grouped history view, and a token-efficient
context loading strategy.

================================================================
SECTION 1 — UI CHANGE: MENU DRAWER INTEGRATION
================================================================

WHAT TO CHANGE:
- Remove the floating popup/bubble AI chat widget from all screens.
- Add a new menu drawer item: "AI Assistant" (with a brain/bot icon).
- Tapping "AI Assistant" opens a full-screen sheet or slide-up modal.

SCREEN STRUCTURE (inside AI Assistant screen):
  - Top tab bar: [New Chat] [History]
  - "New Chat" tab: opens a fresh chat session
  - "History" tab: shows past sessions grouped by date (see Section 4)
  - Back button returns user to wherever they came from in the app

SECTION 2 — FEATURE 1: PERSISTENT AI MEMORY
================================================================

GOAL:
The AI remembers things about the user across all sessions
(preferences, past decisions, recurring topics) without sending
full chat logs every time.

HOW IT WORKS:
- A table called `user_memory` stores one compressed text summary
  per user (max ~150 tokens).
- After every chat session ends (or after every 5 messages),
  a background job calls the AI to update this memory summary.
- On every new AI request, this memory summary is injected into
  the system prompt automatically.

BACKEND TASKS:
1. Create `user_memory` table (see Section 7 for full schema).
2. Create background job: POST /internal/memory/update
   - Input: user_id, recent messages
   - Calls AI with prompt: "Update the user memory summary below
     based on the new conversation. Keep it under 150 tokens.
     Focus on preferences, facts, and recurring patterns.
     Current memory: {existing_summary}
     New messages: {recent_messages}
     Return ONLY the updated summary text."
   - Saves result to user_memory.memory_summary
3. Trigger this job: after session ends OR every 5 messages.

SYSTEM PROMPT INJECTION (add to every AI call):
  ## What you know about this user:
  {user_memory_summary}
  Use this context to personalise responses.
  Do not repeat this information back to the user.

TOKEN BUDGET: ~150 tokens per request.

================================================================
SECTION 3 — FEATURE 2: CHAT HISTORY WITH RESUME
================================================================

GOAL:
User can see a list of all past AI conversations and tap any one
to continue it. Backend sends only a summary (not the full chat
log) to keep token usage low.

BACKEND TASKS:
1. Every session has a `summary` TEXT field in `chat_sessions`.
2. After a session ends or goes idle (15 min), run a background job:
   - Call AI: "Summarize this conversation in under 300 tokens.
     Focus on: what the user was trying to do, what was resolved,
     any open items. Messages: {messages}"
   - Save result to chat_sessions.summary
3. API endpoint: GET /chat/sessions
   - Returns list: [{ id, title, started_at, summary, message_count }]
   - Does NOT return full messages
4. API endpoint: GET /chat/sessions/{id}/resume
   - Returns: { summary, last_10_messages }
   - NEVER returns full message history to the AI context

FRONTEND TASKS:
1. When user taps a past session:
   - Load summary + last 10 messages via GET /chat/sessions/{id}/resume
   - Display last 10 messages in the chat UI as normal
   - Show a small banner at the top: "Resuming chat from {date}"
2. User continues typing — new messages are added to the same session.

CONTEXT SENT TO AI ON RESUME:
  System prompt includes:
  ## Previous conversation summary:
  {session.summary}
  Continue this conversation naturally. The user is resuming
  a previous chat. Do not repeat what was already discussed
  unless asked.

TOKEN BUDGET: ~300 tokens for session summary.

SECTION 4 — FEATURE 3: AUTO MEETING SUMMARY ON MEETING END
================================================================

GOAL:
When a meeting ends, automatically generate a structured summary.
User can then open the AI assistant and ask questions about that
specific meeting.

TRIGGER:
- Hook into the existing "meeting end" event/webhook in Antigravity.
- On this event: fire POST /internal/meeting/generate-summary

BACKEND JOB — POST /internal/meeting/generate-summary:
  Input: { meeting_id, raw_transcript, meeting_title, date, attendees }

  Step 1: Call AI with this prompt:
  -------------------------------------------------------
  Summarize this meeting transcript in structured JSON.
  Return ONLY valid JSON, no explanation, no markdown.

  {
    "title": "meeting title",
    "date": "YYYY-MM-DD",
    "duration_minutes": 0,
    "attendees": ["name1", "name2"],
    "agenda_items": ["item1", "item2"],
    "key_decisions": ["decision1", "decision2"],
    "action_items": [
      { "owner": "name", "task": "description", "due": "YYYY-MM-DD or null" }
    ],
    "brief_summary": "2-3 sentence plain English overview of the meeting."
  }

  Transcript:
  {raw_transcript}
  -------------------------------------------------------

  Step 2: Parse the JSON response.
  Step 3: Save to `meeting_summaries` table.
  Step 4: Create a new `chat_session` with:
    - session_type = 'meeting'
    - meeting_id = {meeting_id}
    - title = meeting_title
    - summary = brief_summary from JSON
  Step 5: Send push notification to attendees:
    "Summary for '{meeting_title}' is ready. Tap to ask questions."

FRONTEND:
- Meeting summary card in history shows: title, date, attendee count,
  brief_summary preview.
- Tapping opens AI chat scoped to that meeting (see Section 5).

SECTION 5 — FEATURE 4: DATE-GROUPED HISTORY VIEW
================================================================

GOAL:
History screen shows sessions grouped by date. Each date is a
section header showing how many meetings/chats happened that day.
Tapping a meeting opens an AI chat scoped ONLY to that meeting.

FRONTEND — History Screen Layout:
  [History tab active]

  13 Aug 2026  ·  2 meetings
  ┌─────────────────────────────────┐
  │ 📋 Team Standup                 │
  │ 10:00 AM · 6 attendees          │
  │ "Discussed sprint blockers..."  │
  └─────────────────────────────────┘
  ┌─────────────────────────────────┐
  │ 📋 Product Review               │
  │ 2:30 PM · 4 attendees           │
  │ "Reviewed Q3 roadmap items..."  │
  └─────────────────────────────────┘

  12 Aug 2026  ·  1 chat
  ┌─────────────────────────────────┐
  │ 💬 General Chat                 │
  │ 4:15 PM · 12 messages           │
  │ "Asked about project status..." │
  └─────────────────────────────────┘

BACKEND:
- GET /chat/sessions?grouped=true
- Returns sessions sorted by started_at DESC, grouped by DATE(started_at)
- Each session includes: id, title, session_type, started_at,
  summary (first 100 chars), message_count, meeting_id (if applicable)

WHEN USER TAPS A MEETING SESSION:
- FE passes context_type: "meeting" to the chat screen
- FE loads: GET /chat/sessions/{id}/meeting-context
  Returns: { meeting_summary_json, session_id, meeting_title, meeting_date }
- AI is scoped ONLY to that meeting's data (see context prompt below)

MEETING-SCOPED AI CONTEXT PROMPT:
  ## Meeting context — {meeting_date}, {meeting_title}
  Attendees: {attendees joined by comma}
  Duration: {duration_minutes} minutes

  Agenda: {agenda_items as numbered list}
  Key decisions: {key_decisions as numbered list}
  Action items: {action_items formatted as "Owner: task (due date)"}
  Summary: {brief_summary}

  IMPORTANT: Answer questions ONLY about this meeting.
  Do not reference other meetings unless the user explicitly asks.
  If information is not in the summary above, say:
  "I don't have that detail from the meeting summary."

TOKEN BUDGET: ~200 tokens for meeting context.

SECTION 6 — FEATURE 5: SUMMARY-ONLY CONTEXT (TOKEN SAVER)
================================================================

GOAL:
Backend never sends full chat history to the AI. Only compressed
summaries are used as context. Full transcripts are stored in DB
for future reference but never injected into AI prompts.

RULES:
1. full chat_messages records are stored in DB — always.
2. Only send to AI: session summary + last 10 messages (for active chat)
   OR session summary only (for history resume context).
3. meeting_summaries JSON is the only meeting data sent to AI.
4. Raw transcripts are stored but NEVER sent to Claude API.

TOTAL TOKEN BUDGET PER REQUEST:
  User memory summary:      ~150 tokens
  Session/meeting context:  ~300 tokens
  Last 10 messages:         ~500–800 tokens (typical)
  Base system prompt:       ~100 tokens
  ─────────────────────────────────────
  Total overhead:           ~1,250 tokens max
  Leaves ample space for user message + AI response.

SUMMARY UPDATE STRATEGY:
- After every session ends: run async job to regenerate session summary.
- After every 5 new messages in an active session: re-run summary job
  in background (do not block the user).
- User memory: update once per session end, not per message.

SECTION 7 — DATABASE SCHEMA
================================================================

TABLE: chat_sessions
  id                UUID PRIMARY KEY
  user_id           UUID NOT NULL (FK → users)
  meeting_id        UUID NULLABLE (FK → meetings)
  title             VARCHAR(255)
  session_type      ENUM('free', 'meeting') DEFAULT 'free'
  started_at        TIMESTAMP NOT NULL
  ended_at          TIMESTAMP NULLABLE
  summary           TEXT NULLABLE        -- compressed, ≤300 tokens
  summary_updated_at TIMESTAMP NULLABLE
  message_count     INT DEFAULT 0
  created_at        TIMESTAMP DEFAULT NOW()

TABLE: chat_messages
  id                UUID PRIMARY KEY
  session_id        UUID NOT NULL (FK → chat_sessions)
  role              ENUM('user', 'assistant') NOT NULL
  content           TEXT NOT NULL
  created_at        TIMESTAMP DEFAULT NOW()
  -- NOTE: This table is for storage/reference only.
  -- Never send full content to AI in history mode.
  -- Only last 10 messages of active session go to AI.

TABLE: meeting_summaries
  id                UUID PRIMARY KEY
  meeting_id        UUID NOT NULL (FK → meetings)
  session_id        UUID NULLABLE (FK → chat_sessions)
  raw_transcript    TEXT             -- stored, never sent to AI
  summary_json      JSONB NOT NULL   -- structured summary (see schema below)
  brief_summary     TEXT             -- plain text, used in session.summary
  generated_at      TIMESTAMP DEFAULT NOW()
  model_used        VARCHAR(100)     -- e.g. 'claude-sonnet-4-20250514'

  -- summary_json structure:
  -- {
  --   "title": string,
  --   "date": string,
  --   "duration_minutes": number,
  --   "attendees": string[],
  --   "agenda_items": string[],
  --   "key_decisions": string[],
  --   "action_items": [{ "owner": string, "task": string, "due": string|null }],
  --   "brief_summary": string
  -- }

TABLE: user_memory
  id                UUID PRIMARY KEY
  user_id           UUID NOT NULL UNIQUE (FK → users)
  memory_summary    TEXT             -- rolling distilled memory, ≤150 tokens
  last_updated      TIMESTAMP DEFAULT NOW()
  version           INT DEFAULT 1    -- increment on each update

SECTION 8 — API ENDPOINTS
================================================================

--- CHAT ---

POST /chat/sessions
  Creates a new chat session.
  Body: { user_id, session_type?, meeting_id? }
  Returns: { session_id, title, started_at }

GET /chat/sessions
  Lists all sessions for the authenticated user.
  Query params: grouped=true (groups by date), session_type filter
  Returns: sessions array (summary preview, no full messages)

GET /chat/sessions/:id/resume
  Loads a session for resumption.
  Returns: { summary, last_10_messages: Message[] }
  IMPORTANT: Returns last 10 messages only, never full history.

GET /chat/sessions/:id/meeting-context
  Loads meeting-scoped context for AI.
  Returns: { meeting_summary_json, meeting_title, meeting_date, session_id }

POST /chat/message
  Sends a message and gets AI response.
  Body: { session_id, message, context_type? ('free'|'meeting') }
  Steps (backend):
    1. Save user message to chat_messages
    2. Load user_memory.memory_summary
    3. Load chat_sessions.summary (if resuming)
    4. If context_type = 'meeting': load meeting_summaries.summary_json
    5. Build system prompt (see Section 9)
    6. Call Claude API with system + last 10 messages
    7. Stream response to client
    8. Save assistant message to chat_messages
    9. Increment session.message_count
    10. If message_count % 5 == 0: trigger async summary update
  Returns: streaming text response

DELETE /chat/sessions/:id
  Soft-deletes a session (set deleted_at).

--- MEETING ---

POST /internal/meeting/generate-summary
  Triggered by meeting end event (internal/webhook).
  Body: { meeting_id, raw_transcript, meeting_title, date, attendees }
  Steps:
    1. Call Claude to generate summary_json
    2. Save to meeting_summaries
    3. Create chat_session (type=meeting)
    4. Send push notification
  Returns: { session_id, summary_json }

--- MEMORY ---

POST /internal/memory/update (internal job, not user-facing)
  Body: { user_id, recent_messages }
  Updates user_memory.memory_summary via AI.

SECTION 13 — IMPORTANT RULES FOR DEVELOPERS
================================================================

1. NEVER send full chat_messages history to the Claude API.
   Only: summary + last 10 messages. This is non-negotiable for
   token and cost efficiency.

2. NEVER send raw_transcript to the Claude API after the summary
   is generated. The transcript is for storage only.

3. ALL AI calls must go through the central system prompt builder
   function. Do not build prompts ad-hoc in individual endpoints.

4. Session summaries and user memory must be updated ASYNC —
   never block the user's message for a summary update.

5. meeting_id on chat_sessions must be set when session_type = 'meeting'.
   Free chat sessions should always have meeting_id = NULL.

6. When a user is in meeting-scoped chat, do NOT inject user_memory
   or general session summary. Only the meeting context prompt goes in.
   (This keeps the AI focused and prevents cross-contamination.)
